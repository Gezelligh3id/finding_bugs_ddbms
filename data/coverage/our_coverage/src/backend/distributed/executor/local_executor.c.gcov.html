       2             :  * local_executor.c
       3             :  *
       4             :  * The scope of the local execution is locally executing the queries on the
       5             :  * shards. In other words, local execution does not deal with any local tables
       6             :  * that are not on shards on the node that the query is being executed. In that
       7             :  * sense, the local executor is only triggered if the node has both the metadata
       8             :  * and the shards (e.g., only Citus MX worker nodes).
       9             :  *
      10             :  * The goal of the local execution is to skip the unnecessary network round-trip
      11             :  * happening on the node itself. Instead, identify the locally executable tasks
      12             :  * and simply call PostgreSQL's planner and executor.
      13             :  *
      14             :  * The local executor is an extension of the adaptive executor. So, the executor
      15             :  * uses adaptive executor's custom scan nodes.
      16             :  *
      17             :  * One thing to note is that Citus MX is only supported with replication factor
      18             :  * to be equal to 1, so keep that in mind while continuing the comments below.
      19             :  *
      20             :  * On the high level, there are 3 slightly different ways of utilizing local
      21             :  * execution:
      22             :  *
      23             :  * (1) Execution of local single shard queries of a distributed table
      24             :  *
      25             :  *      This is the simplest case. The executor kicks at the start of the adaptive
      26             :  *      executor, and since the query is only a single task the execution finishes
      27             :  *      without going to the network at all.
      28             :  *
      29             :  *      Even if there is a transaction block (or recursively planned CTEs), as
      30             :  *      long as the queries hit the shards on the same node, the local execution
      31             :  *      will kick in.
      32             :  *
      33             :  * (2) Execution of local single queries and remote multi-shard queries
      34             :  *
      35             :  *      The rule is simple. If a transaction block starts with a local query
      36             :  *      execution,
      37             :  *      all the other queries in the same transaction block that touch any local
      38             :  *      shard have to use the local execution. Although this sounds restrictive,
      39             :  *      we prefer to implement it in this way, otherwise we'd end-up with as
      40             :  *      complex scenarios as we have in the connection managements due to foreign
      41             :  *      keys.
      42             :  *
      43             :  *      See the following example:
      44             :  *      BEGIN;
      45             :  *          -- assume that the query is executed locally
      46             :  *          SELECT count(*) FROM test WHERE key = 1;
      47             :  *
      48             :  *          -- at this point, all the shards that reside on the
      49             :  *          -- node is executed locally one-by-one. After those finishes
      50             :  *          -- the remaining tasks are handled by adaptive executor
      51             :  *          SELECT count(*) FROM test;
      52             :  *
      53             :  *
      54             :  * (3) Modifications of reference tables
      55             :  *
      56             :  *      Modifications to reference tables have to be executed on all nodes. So,
      57             :  *      after the local execution, the adaptive executor keeps continuing the
      58             :  *      execution on the other nodes.
      59             :  *
      60             :  *      Note that for read-only queries, after the local execution, there is no
      61             :  *      need to kick in adaptive executor.
      62             :  *
      63             :  * (4) Execution of multi shards local queries and
      64             :  *     remote multi-shard queries within a transaction block
      65             :  *
      66             :  *      We prefer local execution when we are inside a transaction block, because not using
      67             :  *      local execution might create some limitations for other commands in the transaction
      68             :  *      block. To simplify things, whenever we are inside a transaction block, we prefer local
      69             :  *      execution if possible.
      70             :  *
      71             :  *  There are also a few limitations/trade-offs that are worth mentioning.
      72             :  *  - The local execution on multiple shards might be slow because the execution
      73             :  *  has to happen one task at a time (e.g., no parallelism).
      74             :  *  - Related with the previous item, COPY command cannot be mixed with local
      75             :  *  execution in a transaction. The implication of that is any part of INSERT..SELECT
      76             :  *  via coordinator cannot happen via the local execution.
      77             :  */
